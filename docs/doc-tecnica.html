⚙ Vision General de la Arquitectura

PHPAdmin Query Manager es una aplicacion PHP de pagina unica (SPA) que permite a los usuarios gestionar conexiones a bases de datos, ejecutar consultas SQL y explorar esquemas de forma visual. La aplicacion sigue un patron MVC simplificado con un backend tipo REST API y un frontend construido en JavaScript puro (sin frameworks).

🔌 Componentes Clave

Router (api/index.php)

El router analiza la URL de la solicitud y el metodo HTTP para despachar la peticion al controlador correspondiente. Utiliza un enfoque basado en patrones de ruta simples. 

// Ejemplo de enrutamiento simplificado
$uri = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
$method = $_SERVER['REQUEST_METHOD'];

switch(true) {
    case str_contains($uri, '/api/connections'):
        $controller = new ConnectionController();
        break;
    case str_contains($uri, '/api/query'):
        $controller = new QueryController();
        break;
    // ... mas rutas
}

Bootstrap (api/bootstrap.php)

El archivo de arranque se encarga de la inicializacion de la aplicacion:

  • Configuracion de la sesion PHP (HttpOnly, SameSite, Secure)
  • Generacion y validacion de token CSRF
  • Establecimiento de cabeceras HTTP de seguridad
  • Supresion de warnings de extensiones no disponibles (E_ALL & ~E_WARNING) para evitar errores fatales cuando pdo_sqlsrv esta en php.ini pero el ODBC Driver no esta instalado
  • Inicializacion del SQLiteManager con auto-migraciones
  • Creacion del usuario por defecto si no existe ninguno

Soporte PWA

La aplicacion es una Progressive Web App (PWA) completa con:

  • Service Worker: estrategia cache-first para activos estaticos (CSS, JS, iconos) y network-first para la API
  • Manifest: permite la instalacion en dispositivos (standalone display)
  • Offline: la interfaz funciona sin conexion (los datos cacheados se muestran, las llamadas a la API retornan 503)

🗃 Base de Datos Interna (SQLite)

La aplicacion utiliza SQLite como base de datos interna para almacenar la configuracion, conexiones, usuarios y registros de auditoria. El archivo se ubica en data/app.sqlite.

ℹ
Nota: SQLite es la base de datos interna de la aplicacion. Las bases de datos destino (MySQL, SQL Server) se configuran como conexiones externas y se acceden mediante PDO.

Sistema de Auto-Migraciones

El SQLiteManager implementa un sistema de migraciones automaticas basado en versiones. Al iniciar la aplicacion, verifica la schema_version (via PRAGMA user_version) y ejecuta las migraciones pendientes de forma secuencial. 

// Logica de migracion simplificada
$currentVersion = $db->query('PRAGMA user_version')->fetchColumn();
$migrations = [1 => 'migrateV1', 2 => 'migrateV2', /* ... */];

foreach ($migrations as $version => $method) {
    if ($version > $currentVersion) {
        $this->$method($db);
        $db->exec("PRAGMA user_version = $version");
    }
}

🔐 Almacenamiento de Credenciales

Contrasenas de Conexiones (AES-256-CBC)

Las contrasenas de las conexiones a bases de datos externas se almacenan cifradas usando AES-256-CBC a traves del EncryptionService. El valor cifrado incluye el vector de inicializacion (IV) concatenado con el texto cifrado, codificado en Base64.

Contrasenas de Usuarios (bcrypt)

Las contrasenas de los usuarios del sistema se almacenan como hashes irreversibles usando PASSWORD_BCRYPT de PHP con costo por defecto. Se verifican con password_verify().

Conexiones BD Externas Usuarios del Sistema | | v v AES-256-CBC (reversible) bcrypt (irreversible) | | v v password_encrypted password_hash (se descifra para conectar) (se verifica con password_verify)

🔒 Mecanismos de Seguridad

PHPAdmin Query Manager implementa multiples capas de seguridad para proteger los datos y las operaciones de la aplicacion.

Autenticacion Basada en Sesiones

La autenticacion utiliza sesiones nativas de PHP con las siguientes configuraciones de seguridad:

  • HttpOnly: la cookie de sesion no es accesible desde JavaScript
  • SameSite=Strict: previene envio de cookies en solicitudes cross-site
  • Secure: solo se envia por HTTPS (cuando esta habilitado)
  • Strict mode: PHP solo acepta IDs de sesion generados por el servidor
session_set_cookie_params([
    'httponly'  => true,
    'samesite' => 'Strict',
    'secure'   => $isHttps,
]);
ini_set('session.use_strict_mode', 1);

Hashing de Contrasenas

Las contrasenas de usuario se hashean con bcrypt (PASSWORD_BCRYPT), el algoritmo recomendado por PHP para almacenamiento seguro de contrasenas. 

// Almacenamiento
$hash = password_hash($password, PASSWORD_BCRYPT);

// Verificacion
if (password_verify($inputPassword, $storedHash)) {
    // Autenticacion exitosa
}

Proteccion CSRF

Cada sesion genera un token CSRF unico usando random_bytes(32). Todas las solicitudes que modifican datos (POST, PUT, DELETE) deben incluir el token en la cabecera X-CSRF-Token. 

// Generacion del token (bootstrap.php)
if (!isset($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}

// Validacion en solicitudes mutantes
if (in_array($method, ['POST', 'PUT', 'DELETE'])) {
    $token = $_SERVER['HTTP_X_CSRF_TOKEN'] ?? '';
    if (!hash_equals($_SESSION['csrf_token'], $token)) {
        // 403 Forbidden
    }
}
ℹ
El token se regenera en cada inicio de sesion para prevenir ataques de fijacion de sesion.

Cifrado de Contrasenas de Conexion

Las contrasenas de las conexiones a bases de datos externas se cifran con AES-256-CBC para poder ser descifradas al momento de establecer la conexion PDO.

Fuente de la clave de cifrado (por prioridad):

  1. Variable de entorno PHPADMIN_ENCRYPTION_KEY
  2. Archivo config/.encryption_key
  3. Si ninguna existe, se genera automaticamente un archivo con una clave aleatoria
// Cifrado
$iv = openssl_random_pseudo_bytes(16);
$encrypted = openssl_encrypt($data, 'AES-256-CBC', $key, 0, $iv);
return base64_encode($iv . $encrypted);

// Descifrado
$data = base64_decode($encryptedData);
$iv = substr($data, 0, 16);
$ciphertext = substr($data, 16);
return openssl_decrypt($ciphertext, 'AES-256-CBC', $key, 0, $iv);
⚠
Critico: La clave de cifrado debe mantenerse segura. Si se pierde, todas las contrasenas almacenadas seran irrecuperables. Respaldela en un lugar seguro.

Proteccion de Directorios y Archivos

Las reglas de .htaccess en la raiz del proyecto bloquean el acceso directo a archivos y directorios sensibles: 

# Bloquear acceso a directorios sensibles
RewriteRule ^data/ - [F,L]
RewriteRule ^config/ - [F,L]
RewriteRule ^sql/ - [F,L]
RewriteRule ^docs/ - [F,L]

# Bloquear archivos SQLite y ocultos
RewriteRule \.sqlite$ - [F,L]
RewriteRule (^|/)\.(?!well-known) - [F,L]

Ademas, los directorios config/ y data/ contienen su propio .htaccess con Deny from all.

CORS (Cross-Origin Resource Sharing)

La API esta restringida al mismo origen. No se configuran cabeceras Access-Control-Allow-Origin para origenes externos, lo que previene solicitudes cross-origin no autorizadas.

Seguridad de la API

🔒 requireAuth()

Verifica que el usuario tenga una sesion activa. Retorna 401 si no esta autenticado.

👑 requireAdmin()

Requiere rol de administrador. Usado en rutas de gestion de usuarios. Retorna 403 si no es admin.

🔓 checkPasswordChange()

Bloquea todas las operaciones hasta que el usuario cambie su contrasena temporal.

🛠 DEBUG_MODE

En produccion (false), los errores no exponen stack traces ni detalles internos.

Politica de Contrasenas

  • Longitud minima: 4 caracteres
  • Cambio obligatorio: el flag must_change_password fuerza el cambio en el primer inicio de sesion
  • Usuario por defecto: al instalar, se crea un usuario admin con contrasena aleatoria y must_change_password = 1
⚠
Primer acceso: La contrasena generada aleatoriamente se muestra una unica vez en la consola del servidor o en los logs. El administrador debe cambiarla inmediatamente.

⚙ Capa de Servicios

Los servicios encapsulan la logica de negocio de la aplicacion. Cada servicio es una clase PHP ubicada en api/Services/ con responsabilidades bien definidas.

Controller | +-- AuthService (autenticacion y usuarios) +-- DatabaseService (conexiones PDO) +-- QueryExecutionService (ejecucion SQL + auditoria) +-- SchemaService (introspeccion de esquemas) +-- AuditService (registros de auditoria) +-- EncryptionService (cifrado AES-256-CBC) +-- XlsxService (generacion Excel) +-- CrossJoinService (JOIN en memoria entre result sets) +-- VirtualQueryEngine (mini SQL engine en memoria) +-- SchemaCompareService (comparacion de esquemas + DDL)

🔐 EncryptionService

Cifrado y descifrado de contrasenas de conexion usando AES-256-CBC.

  • encrypt($data) Cifra un texto plano. Genera un IV aleatorio de 16 bytes y retorna IV+cifrado en Base64.
  • decrypt($data) Descifra datos previamente cifrados. Extrae el IV de los primeros 16 bytes.
  • getKey() Obtiene la clave: primero busca la variable de entorno PHPADMIN_ENCRYPTION_KEY, luego el archivo config/.encryption_key. Si ninguna existe, genera una clave aleatoria. 
// Uso tipico en ConnectionController
$encrypted = $encryptionService->encrypt($plainPassword);
// Guardar $encrypted en connections.password_encrypted

// Al conectar
$password = $encryptionService->decrypt($connection['password_encrypted']);

👤 AuthService

Gestion de autenticacion, sesiones y usuarios del sistema.

  • ensureDefaultUser() Crea el usuario admin si no existe ningun usuario. Genera contrasena aleatoria con must_change_password = 1.
  • login($username, $password) Verifica credenciales con bcrypt. Inicia sesion PHP y regenera CSRF token.
  • logout() Destruye la sesion y elimina la cookie.
  • isLoggedIn() Verifica si existe una sesion activa.
  • getCurrentUser() Retorna los datos del usuario autenticado actual.
  • changePassword($userId, $current, $new) Cambia la contrasena del usuario. Limpia el flag must_change_password.
  • createUser($data) Crea un nuevo usuario (solo admin). Hashea la contrasena con bcrypt.
  • updateUser($id, $data) Actualiza datos del usuario.
  • deleteUser($id) Elimina un usuario. Previene eliminar el ultimo administrador.
⚠
Proteccion: El sistema impide eliminar el ultimo usuario con rol admin para evitar bloqueo total de la aplicacion.

🗂 DatabaseService

Fabrica de conexiones PDO con cache y soporte multi-driver.

  • connect($connectionConfig) Crea una conexion PDO a partir de la configuracion. Construye el DSN segun el driver (mysql o sqlsrv).
  • getConnection($id) Obtiene una conexion cacheada o crea una nueva. Descifra la contrasena con EncryptionService.
  • testConnection($config) Prueba la conectividad sin almacenar. Util para validar antes de guardar.
  • buildDsn($config) Construye la cadena DSN segun el driver:
  • createPdo($dsn, $user, $pass) Valida que el driver solicitado este realmente disponible antes de crear la conexion PDO. Si el driver no esta funcional, lanza una excepcion con mensaje claro en espanol (ej: "El driver sqlsrv no esta disponible en este servidor"). 
// MySQL
"mysql:host={$host};port={$port};dbname={$db};charset={$charset}"

// SQL Server
"sqlsrv:Server={$host},{$port};Database={$db}"
ℹ
Validacion de drivers (v1.1.0): El endpoint /csrf-token ahora retorna la lista de drivers disponibles. El backend valida la disponibilidad real del driver intentando una conexion dummy con new PDO('sqlsrv:Server=__test__') y verificando si el error contiene "ODBC". Que pdo_sqlsrv este cargado en PHP no garantiza que el Microsoft ODBC Driver este instalado en el sistema operativo.

▶ QueryExecutionService

Motor de ejecucion de consultas SQL con auditoria integrada.

  • executeDirect($connection, $sql, $db) Ejecuta SQL directamente contra la base de datos destino. Registra la ejecucion en audit_logs incluyendo tiempo, filas y estado.
  • executeStoredProcedure($connection, $sql, $db) Ejecuta la consulta a traves del stored procedure configurado (sp_name) que retorna resultados en formato JSON. Soporta MySQL y SQL Server.
  • formatError($exception, $driver) Formatea mensajes de error segun el driver para presentacion amigable al usuario.
ℹ
El modo stored procedure utiliza procedimientos almacenados especificos (ver sql/sp_mysql_json_query.sql y sql/sp_sqlserver_json_query.sql) que encapsulan la ejecucion y retornan los resultados como JSON, permitiendo mejor control de errores y limites de filas.

🔍 SchemaService

Introspeccion de esquemas de base de datos con soporte multi-driver.

  • getDatabases($connection) Lista las bases de datos disponibles en el servidor.
  • getTables($connection, $db) Lista las tablas de una base de datos.
  • getViews($connection, $db) Lista las vistas de una base de datos.
  • getProcedures($connection, $db) Lista los procedimientos almacenados.
  • getFunctions($connection, $db) Lista las funciones definidas por el usuario.
  • getColumns($connection, $db, $table) Obtiene las columnas de una tabla con tipo, nulabilidad, clave y valor por defecto.
  • getRoutineParams($connection, $db, $routine) Lista los parametros de un procedimiento o funcion.
  • getRoutineDefinition($connection, $db, $routine) Obtiene el codigo fuente de un procedimiento o funcion.
ℹ
Cada metodo contiene SQL diferente para MySQL y SQL Server, ya que los catologos de sistema (INFORMATION_SCHEMA, sys.*) difieren significativamente entre ambos motores.

🔗 CrossJoinService

Servicio para realizar operaciones JOIN y operaciones de conjuntos entre result sets en memoria. Ubicado en api/Services/CrossJoinService.php.

  • join($leftData, $rightData, $type, $leftKey, $rightKey) Realiza un JOIN en memoria entre dos result sets. Soporta tipos: INNER, LEFT, RIGHT y CROSS.
  • chainJoin($datasets, $joinConfigs) Ejecuta una cadena secuencial de N operaciones JOIN, donde el resultado de cada JOIN alimenta al siguiente.
  • setOperation($datasets, $operation) Ejecuta operaciones de conjuntos entre N datasets: UNION, UNION_ALL, INTERSECT y EXCEPT.
⚠
Limites de memoria: MAX_ROWS = 10,000 filas por dataset de entrada. MAX_CROSS_ROWS = 50,000 filas para el resultado de un CROSS JOIN. Estos limites previenen el agotamiento de memoria del servidor.

⚙ VirtualQueryEngine

Mini motor SQL que ejecuta consultas contra datasets en memoria. Implementa un pipeline completo de Tokenizer, Parser (descenso recursivo) y Ejecutor sobre un AST. Ubicado en api/Services/VirtualQueryEngine.php (1630 lineas).

Pipeline de ejecucion:

SQL String | +-- Tokenizer (analisis lexico) | Genera tokens: keywords, identificadores, literales, operadores | +-- Parser (descenso recursivo) | Construye AST (Abstract Syntax Tree) | +-- Executor Recorre el AST y opera sobre arrays PHP en memoria

Clausulas SQL soportadas:

  • SELECT (con alias, *, expresiones)
  • FROM (con alias de tabla)
  • JOIN (INNER, LEFT, RIGHT, CROSS) con ON
  • WHERE (todos los operadores: =, !=, <, >, <=, >=, AND, OR, NOT, IN, BETWEEN, IS NULL, IS NOT NULL)
  • ORDER BY (ASC/DESC, multiples columnas)
  • LIMIT / OFFSET
  • GROUP BY + HAVING
  • DISTINCT
  • UNION / INTERSECT / EXCEPT

Funciones integradas:

  • Agregacion: COUNT, SUM, AVG, MAX, MIN
  • Cadenas: UPPER, LOWER, TRIM, CONCAT
  • Patrones: LIKE (convierte patrones SQL % y _ a regex)
ℹ
Semantica NULL: El motor sigue la semantica SQL estandar para NULL: comparaciones con NULL retornan NULL (no true/false), IS NULL / IS NOT NULL son las unicas formas validas de testear nulidad. MAX_RESULT_ROWS = 50,000.

📋 SchemaCompareService

Servicio de comparacion de esquemas entre dos bases de datos, con generacion de scripts DDL de migracion. Ubicado en api/Services/SchemaCompareService.php (936 lineas). Soporta comparaciones cross-driver (MySQL vs SQL Server).

  • compareAll($sourceConn, $targetConn, $sourceDb, $targetDb) Comparacion completa del esquema: tablas, vistas, procedimientos almacenados y funciones. Retorna un informe detallado con diferencias categorizadas.
  • compareTables($source, $target) Compara la lista de tablas entre origen y destino. Identifica tablas que solo existen en uno de los dos lados.
  • compareTableColumns($source, $target, $table) Comparacion a nivel de columna: detecta diferencias en tipo de dato, nulabilidad, valor por defecto, columnas agregadas y columnas eliminadas.
  • compareViews($source, $target) Compara definiciones de vistas entre ambas bases de datos.
  • compareProcedures($source, $target) Compara definiciones de procedimientos almacenados.
  • compareFunctions($source, $target) Compara definiciones de funciones de usuario.
  • generateScript($differences, $driver) Genera un script DDL de migracion a partir del informe de diferencias. Produce sintaxis correcta para MySQL y SQL Server (CREATE TABLE, ALTER TABLE, ADD/DROP/MODIFY COLUMN, CREATE/ALTER VIEW, etc.).
ℹ
Mapeo cross-driver: El servicio normaliza los tipos de datos entre MySQL y SQL Server (por ejemplo, INT vs int, VARCHAR vs nvarchar) para evitar falsos positivos en la comparacion.

API Client (api-client.js)

Cliente centralizado para todas las comunicaciones con el backend. Encapsula la logica de autenticacion, CSRF y manejo de errores.

Caracteristicas principales:

  • CSRF automatico: incluye el token X-CSRF-Token en todas las solicitudes mutantes
  • AbortController: permite cancelar solicitudes en curso cuando el usuario navega a otro panel
  • Redireccion de autenticacion: si la API retorna 401, redirige automaticamente al login
  • Manejo de errores: parsea respuestas de error y muestra mensajes amigables

Progressive Web App (PWA)

Service Worker (service-worker.js)

El Service Worker implementa dos estrategias de cache diferentes segun el tipo de recurso:

📦 Cache-First

Para activos estaticos (CSS, JS, iconos, fuentes). Sirve desde cache si esta disponible, actualiza en segundo plano.

🌐 Network-First

Para llamadas a la API (/api/*). Intenta la red primero; si falla, retorna respuesta 503 con mensaje offline. 

// Estrategia de cache en el Service Worker
self.addEventListener('fetch', (event) => {
    if (event.request.url.includes('/api/')) {
        // Network-first para API
        event.respondWith(
            fetch(event.request)
                .catch(() => new Response(
                    '{"error": "Sin conexion"}',
                    { status: 503 }
                ))
        );
    } else {
        // Cache-first para activos estaticos
        event.respondWith(
            caches.match(event.request)
                .then(cached => cached || fetch(event.request))
        );
    }
});
