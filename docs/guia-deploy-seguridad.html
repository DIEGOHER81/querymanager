⚙ Requisitos del Servidor

A continuacion se detallan los requisitos minimos y recomendados para ejecutar Query Manager en un servidor de produccion.

PHP 8.0 o superior Requerido

Query Manager requiere PHP 8.0+ por el uso de named arguments, match expressions y union types.

Extension Uso Estado
pdo Abstraccion de acceso a bases de datos Requerido
pdo_sqlite Base de datos interna de la aplicacion Requerido
pdo_mysql Conexion a bases de datos MySQL / MariaDB Segun uso
pdo_sqlsrv o pdo_dblib Conexion a bases de datos SQL Server Segun uso
openssl Encriptacion de credenciales almacenadas Requerido
json API REST (request/response) Requerido
mbstring Manejo correcto de cadenas multibyte Requerido

Apache con mod_rewrite Requerido

El servidor Apache debe tener habilitado el modulo mod_rewrite y la directiva AllowOverride All para que los archivos .htaccess funcionen correctamente.

ℹ
La mayoria de hostings compartidos ya tienen mod_rewrite habilitado por defecto. Si usas cPanel, esto generalmente ya esta configurado.

Directorios con permisos de escritura Requerido

Los siguientes directorios deben tener permisos de escritura para el proceso de PHP:

  • data/ — Almacena la base de datos SQLite (app.sqlite) y la contrasena inicial
  • config/ — Almacena la clave de encriptacion auto-generada (.encryption_key)

HTTPS (SSL/TLS) Recomendado

Se recomienda encarecidamente usar HTTPS para proteger las credenciales en transito. La mayoria de hostings ofrecen certificados SSL gratuitos via Let's Encrypt.

⚠
Sin HTTPS, las credenciales de login y las cookies de sesion viajan en texto plano. En la seccion de Configuracion Avanzada se explica como forzar HTTPS.

Compatibilidad con Hosting Compartido

Query Manager esta disenado para funcionar en la mayoria de hostings compartidos. Puntos a verificar:

  • Verificar la version de PHP en cPanel → Select PHP Version o PHP Info
  • La extension pdo_sqlite esta disponible en casi todos los hostings
  • Para SQL Server (pdo_sqlsrv), algunos hostings Linux no lo ofrecen; considerar pdo_dblib como alternativa
  • El archivo .htaccess funciona sin configuracion adicional en la mayoria de hostings con cPanel
✔
Probado y compatible con: cPanel, Plesk, DirectAdmin, y la mayoria de hostings con Apache + PHP 8.0+.

🚀 Primer Deploy (Paso a Paso)

Sigue estos pasos en orden para desplegar Query Manager en tu servidor por primera vez.

1
Preparar archivos ▼

Clona o descarga el proyecto en tu equipo local. Antes de subir los archivos, es critico excluir ciertos archivos sensibles.

⚠
NO subir estos archivos al hosting:
Contienen datos sensibles del entorno local que no deben existir en produccion.
  • ✖ data/app.sqlite — Contiene contrasenas locales y logs de auditoria
  • ✖ config/.encryption_key — Clave de encriptacion del entorno local
  • ✖ data/.initial_password — Contrasena inicial (si existe)
  • ✖ *.bak, *.log — Archivos de respaldo y registros
✔
SI subir estos archivos (son seguros):
Todos los archivos .php, .js, .css, .html, .htaccess, manifest.json, iconos y carpetas vacias data/ y config/.

Archivos nuevos en v1.1.0:
api/Controllers/MultiQueryController.php, api/Controllers/SchemaCompareController.php, api/Services/CrossJoinService.php, api/Services/VirtualQueryEngine.php, api/Services/SchemaCompareService.php, assets/js/components/multi-query.js, assets/js/components/cross-join.js, assets/js/components/schema-compare.js, assets/js/components/sql-intellisense.js, assets/js/components/client-export.js
2
Subir archivos al hosting ▼

Sube los archivos al servidor mediante FTP/SFTP (FileZilla, WinSCP) o el Administrador de Archivos de cPanel.

  • Sube a un subdirectorio, por ejemplo: public_html/phpadmin/
  • Asegurate de subir los archivos .htaccess (son archivos ocultos; activa la opcion de ver archivos ocultos en tu cliente FTP)
⚠
IMPORTANTE: Si la ruta de instalacion difiere de /phpadmin/, debes actualizar la directiva RewriteBase en el archivo .htaccess principal. 
# Ejemplo: si instalas en public_html/mi-app/
RewriteBase /mi-app/
3
Configurar permisos ▼

Establece los permisos correctos en los directorios y archivos. Puedes hacerlo via SSH o desde el Administrador de Archivos de cPanel.

# Directorios con escritura para PHP
chmod 755 data/
chmod 755 config/

# Archivos .htaccess (solo lectura)
chmod 644 .htaccess
chmod 644 config/.htaccess
chmod 644 data/.htaccess
ℹ
En cPanel, haz clic derecho sobre el directorio → Change Permissions y establece los valores indicados.
4
Configurar variable de entorno (Recomendado) ▼

La clave de encriptacion se usa para cifrar/descifrar las contrasenas de conexion a bases de datos. Hay tres opciones para configurarla:

Opcion A: Variable de entorno en cPanel Recomendado

En cPanel → Software → Select PHP Version → Options/Environment Variables, agregar:

PHPADMIN_ENCRYPTION_KEY=<tu_clave_base64>

Opcion B: En .htaccess No recomendado

Agregar al .htaccess principal:

SetEnv PHPADMIN_ENCRYPTION_KEY <tu_clave_base64>
⚠
Esta opcion es menos segura porque el .htaccess podria ser leido en ciertas configuraciones de servidor.

Opcion C: Generacion automatica Aceptable

Si no se configura ninguna variable, la aplicacion genera automaticamente un archivo config/.encryption_key. Esto es aceptable siempre que el directorio config/ este correctamente protegido por .htaccess.

Generar una clave

Ejecuta este comando para generar una clave segura:

php -r "echo base64_encode(random_bytes(32));"
5
Primer acceso ▼

Navega a tu instalacion:

https://tudominio.com/phpadmin/

La aplicacion ejecutara automaticamente la inicializacion:

  1. Crea data/app.sqlite con todas las tablas necesarias
  2. Genera una contrasena aleatoria para el usuario admin
  3. Guarda la contrasena en el archivo data/.initial_password
ℹ
Para obtener la contrasena: Accede al archivo data/.initial_password mediante FTP o el Administrador de Archivos de cPanel. Este archivo contiene la contrasena en texto plano.

Proceso de login:

  1. Inicia sesion con usuario admin y la contrasena del archivo
  2. El sistema te obligara a cambiar la contrasena (dialogo SweetAlert2)
  3. Ingresa una nueva contrasena segura
  4. Ya puedes usar la aplicacion
⚠
IMPORTANTE: Elimina el archivo data/.initial_password despues de leerlo. Este archivo contiene la contrasena en texto plano y no debe permanecer en el servidor.
6
Verificar seguridad ▼

Verifica que las protecciones de seguridad estan funcionando correctamente. Cada una de estas URLs debe devolver un error 403 Forbidden:

  • ✔ No se puede acceder a https://dominio.com/phpadmin/data/app.sqlite (debe retornar 403)
  • ✔ No se puede acceder a https://dominio.com/phpadmin/config/.encryption_key (debe retornar 403)
  • ✔ No se puede acceder a https://dominio.com/phpadmin/sql/ (debe retornar 403)
  • ✔ No se puede acceder a https://dominio.com/phpadmin/docs/ (debe retornar 403)
  • ✔ No se puede acceder a https://dominio.com/phpadmin/.env (debe retornar 403)
  • ✔ No se puede acceder a https://dominio.com/phpadmin/.gitignore (debe retornar 403)
  • ✔ La pagina de login carga correctamente
  • ✔ Despues de iniciar sesion, todos los modulos funcionan correctamente
✔
Si todos los puntos estan verificados, tu instalacion esta correctamente protegida.

🌐 Publicacion Paso a Paso en Hosting Compartido

Guia basada en experiencia real desplegando en hosting compartido (cPanel). Sigue estos pasos en orden.

1
Generar el paquete de deploy ▼

Ejecuta el script de deploy desde PowerShell en la raiz del proyecto:

.\deploy.ps1 -BasePath "/" -Production -Zip

Esto genera un directorio deploy/ limpio y un archivo ZIP listo para subir. El script automaticamente:

  • Configura RewriteBase segun el -BasePath indicado
  • Excluye archivos sensibles: app.sqlite, .encryption_key, .gitignore, .initial_password
  • Excluye archivos de desarrollo: deploy.ps1, check_setup.php, docs/, etc.
  • Genera un paquete listo para produccion
ℹ
Si la app estara en un subdirectorio, usa -BasePath "/nombre-directorio/". Si estara en la raiz del dominio, usa -BasePath "/".
2
Subir archivos al hosting ▼

Sube los archivos al servidor via FTP (FileZilla, WinSCP) o cPanel File Manager:

  • Sube TODO el contenido del directorio deploy/ a la raiz del dominio/subdominio
  • Verifica que .htaccess se subio correctamente (es un archivo oculto)
  • En cPanel File Manager: activar "Mostrar archivos ocultos" en Configuracion
3
Verificar permisos ▼ 
# Directorios que necesitan escritura
chmod 755 data/     # o 775 si 755 no funciona
chmod 755 config/   # o 775 si 755 no funciona

# Archivos: solo lectura
# Archivos .php, .js, .css, .html, .htaccess = 644
# Carpetas = 755
4
Verificar .htaccess y RewriteBase ▼
  • Si la app esta en la raiz del dominio: RewriteBase /
  • Si esta en un subdirectorio: RewriteBase /nombre-directorio/
✔
Auto-deteccion en v1.1.0: El archivo api-client.js ahora auto-detecta la ruta base, no necesita configuracion manual. Igualmente, api/index.php auto-detecta el basePath.
5
Primer acceso ▼

Abre la URL en el navegador. El sistema crea automaticamente:

  • data/app.sqlite — Base de datos interna (usuarios, conexiones, auditoria)
  • config/.encryption_key — Clave de encriptacion para contrasenas de conexiones
  • data/.initial_password — Contrasena aleatoria del usuario admin
6
Obtener credenciales iniciales ▼

El archivo data/.initial_password NO es accesible por web (protegido por .htaccess). Para leerlo:

  • Opcion 1: Acceder via FTP y descargar el archivo
  • Opcion 2: En cPanel File Manager, activar "Mostrar archivos ocultos" y navegar a data/.initial_password
  • Opcion 3: Subir check_setup.php (disponible en la raiz del proyecto) a la raiz del hosting y abrir en navegador
⚠
ELIMINAR check_setup.php inmediatamente despues de usarlo. Es una herramienta de diagnostico que no debe permanecer en produccion.
7
Login y cambio de contrasena ▼
  1. Ingresar con usuario: admin y la contrasena del archivo
  2. El sistema obliga a cambiar la contrasena en el primer login
  3. Ingresar una nueva contrasena segura
⚠
Eliminar data/.initial_password despues del primer acceso exitoso.
8
Verificar seguridad post-deploy ▼

Verifica estas URLs en el navegador:

  • ✔ https://dominio.com/data/ devuelve 403 Forbidden
  • ✔ https://dominio.com/config/ devuelve 403 Forbidden
  • ✔ La consola del navegador (F12) no muestra errores 500
  • ✔ El login funciona correctamente

🏗 Compatibilidad con Hosting Compartido

Consideraciones importantes al usar hosting compartido (cPanel, Plesk, etc.).

Bases de Datos Soportadas

⚠
Hosting compartido Linux solo soporta MySQL/MariaDB.
SQL Server requiere un VPS o servidor dedicado con Microsoft ODBC Driver instalado.
  • La app detecta automaticamente los drivers disponibles en tiempo de ejecucion y muestra un error claro si sqlsrv no esta disponible
  • Si el hosting tiene pdo_sqlsrv en php.ini pero sin ODBC Driver instalado, la app suprime el warning automaticamente y excluye el driver de la lista disponible
  • El formulario de conexion muestra solo los drivers realmente funcionales; SQL Server aparece deshabilitado si no esta disponible
  • No se requiere configuracion manual: la deteccion y adaptacion es completamente automatica

Extensiones PHP Requeridas

Extension Requerida Uso
pdo_sqlite Si Base de datos interna (usuarios, conexiones, auditoria)
openssl Si Encriptacion de contrasenas de conexiones
json Si Comunicacion API (JSON)
pdo_mysql Para MySQL Conexion a servidores MySQL/MariaDB
pdo_sqlsrv + ODBC Driver Opcional Conexion a SQL Server (solo VPS/dedicado)

🔒 Seguridad Implementada

Query Manager implementa multiples capas de seguridad para proteger la aplicacion y los datos. A continuacion se detalla cada mecanismo.

Capas de Proteccion

1
Autenticacion por sesion

Sesiones PHP con cookies configuradas: HttpOnly (no accesibles via JavaScript), SameSite=Strict (protege contra CSRF basico), Secure (solo via HTTPS).

2
Proteccion CSRF

Token unico por sesion, regenerado en cada inicio de sesion. Se valida automaticamente en todas las peticiones que modifican datos (POST, PUT, DELETE).

3
Encriptacion de contrasenas de conexion

Las contrasenas de las conexiones a bases de datos se almacenan encriptadas con AES-256-CBC. La clave se obtiene de una variable de entorno o de un archivo auto-generado protegido.

4
Hashing de contrasenas de usuario

Las contrasenas de los usuarios de la aplicacion se almacenan con hash bcrypt (PASSWORD_BCRYPT). No se almacenan en texto plano.

5
Cambio obligatorio de contrasena

En el primer inicio de sesion, el sistema obliga al usuario a cambiar su contrasena. Todas las rutas estan bloqueadas excepto logout, change-password y me hasta que se complete el cambio.

6
Headers HTTP de seguridad

Se envian cabeceras de seguridad en cada respuesta para proteger contra ataques comunes.

7
CORS restrictivo

Solo se permiten peticiones del mismo origen (same-origin). Las peticiones cross-origin son rechazadas.

8
Proteccion de directorios

Reglas .htaccess con directivas Deny from all en los directorios data/, config/, sql/, docs/. Tambien se bloquean archivos .sqlite y archivos ocultos (.*).

9
DEBUG_MODE desactivado

En produccion, el modo de depuracion esta desactivado. No se exponen trazas de pila, rutas de archivos ni mensajes de error detallados en las respuestas de la API.

10
Roles de usuario

Sistema de roles admin vs user. Los endpoints exclusivos de administrador requieren verificacion de rol. Los usuarios regulares solo pueden acceder a las funcionalidades asignadas.

Headers HTTP de Seguridad (Detalle)

Header Valor Proteccion
X-Content-Type-Options nosniff Evita MIME-type sniffing
X-Frame-Options DENY Previene clickjacking (no se puede cargar en iframes)
X-XSS-Protection 1; mode=block Activa filtro XSS del navegador
Referrer-Policy strict-origin-when-cross-origin Limita informacion del referrer en peticiones externas
Content-Security-Policy script-src 'self' cdn.jsdelivr.net Restringe scripts solo a mismo origen y CDN autorizado

🛠 Configuracion Avanzada

Opciones de configuracion para personalizar y optimizar la instalacion en entornos especificos.

Constantes en app.php

El archivo config/app.php contiene las constantes principales de configuracion:

Constante Descripcion Valor por defecto
APP_NAME Nombre de la aplicacion 'PHPAdmin'
APP_VERSION Version actual '1.1.0'
DATA_DIR Directorio de datos __DIR__.'/../data'
CONFIG_DIR Directorio de configuracion __DIR__
SQLITE_DB Ruta a la base de datos SQLite DATA_DIR.'/app.sqlite'
ENCRYPTION_KEY_FILE Archivo de clave de encriptacion CONFIG_DIR.'/.encryption_key'
AUDIT_ENABLED Habilitar/deshabilitar registro de auditoria true
JSON_RESULT_LIMIT Maximo de filas en modo JSON (Stored Procedure) 10
DEFAULT_QUERY_TIMEOUT Timeout de PDO en segundos 30
DEBUG_MODE Modo de depuracion false
ENCRYPTION_KEY_ENV Lee clave de encriptacion de variable de entorno getenv('PHPADMIN_ENCRYPTION_KEY')
⚠
NUNCA establezca DEBUG_MODE en true en produccion. Esto expone trazas de pila, rutas del servidor y otra informacion sensible en las respuestas de error.

Cambiar la Ruta Base

Si la aplicacion no se instala en /phpadmin/, es necesario actualizar la ruta base en tres archivos:

1
.htaccess principal ▼

Cambiar la directiva RewriteBase:

# Antes:
RewriteBase /phpadmin/

# Despues (ejemplo: instalado en /mi-app/):
RewriteBase /mi-app/
2
api/index.php ▼

Actualizar la variable $basePath:

// Antes:
$basePath = '/phpadmin/api';

// Despues:
$basePath = '/mi-app/api';
3
assets/js/api-client.js ▼

Actualizar la propiedad baseUrl:

// Antes:
baseUrl = '/phpadmin/api';

// Despues:
baseUrl = '/mi-app/api';

Forzar HTTPS

Para redirigir automaticamente todo el trafico HTTP a HTTPS, agrega estas lineas al inicio del archivo .htaccess principal:

# Forzar HTTPS
RewriteCond %{HTTPS} off
RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
ℹ
Asegurate de tener un certificado SSL valido antes de activar esta regla. La mayoria de hostings con cPanel ofrecen certificados gratuitos via Let's Encrypt o AutoSSL.

Stored Procedures (Modo de Ejecucion JSON)

Para utilizar el modo de ejecucion JSON, es necesario instalar el Stored Procedure correspondiente en las bases de datos destino:

Motor Archivo SQL Instruccion
MySQL / MariaDB sql/sp_mysql_json_query.sql Ejecutar el script en cada base de datos donde se usara el modo JSON
SQL Server sql/sp_sqlserver_json_query.sql Ejecutar el script en cada base de datos donde se usara el modo JSON

Despues de instalar el SP, configura el nombre del procedimiento en la configuracion de cada conexion dentro de la aplicacion.

ℹ
El limite de filas en modo JSON esta controlado por la constante JSON_RESULT_LIMIT (por defecto: 10 filas). Puedes modificar este valor en config/app.php.

🔧 Solucion de Problemas

Problemas comunes y sus soluciones. Haz clic en cada problema para ver la solucion.

● 403 Forbidden en todas las paginas

Causa: El modulo mod_rewrite no esta habilitado o la directiva AllowOverride no esta configurada como All.

Solucion:

  • Verifica que mod_rewrite esta habilitado: en terminal, ejecuta apache2ctl -M | grep rewrite
  • En el archivo de configuracion de Apache (httpd.conf o virtual host), asegurate de que AllowOverride All esta configurado para el directorio
  • En cPanel, esto generalmente ya esta habilitado. Contacta a tu proveedor si persiste
● 500 Internal Server Error

Causa: Error en la configuracion de PHP o en el archivo .htaccess.

Solucion:

  • Verifica la version de PHP (se requiere 8.0+). En cPanel: Select PHP Version
  • Revisa el archivo de log de errores de Apache: error_log (generalmente en la raiz de public_html o en /var/log/apache2/)
  • Verifica que el archivo .htaccess no tiene directivas no soportadas por tu hosting
  • Asegurate de que las extensiones PHP requeridas estan habilitadas
● No se puede conectar a MySQL / SQL Server

Causa: El hosting puede no permitir conexiones salientes a bases de datos externas, o el firewall bloquea la conexion.

Solucion:

  • Verifica que el hosting permite conexiones salientes al puerto del servidor de base de datos (3306 para MySQL, 1433 para SQL Server)
  • Si la base de datos esta en otro servidor, verifica las reglas del firewall y whitelist la IP del hosting
  • Verifica que la extension PDO correspondiente esta instalada (pdo_mysql o pdo_sqlsrv / pdo_dblib)
  • Prueba la conexion desde PHP con un script simple para aislar el problema
● CSRF token invalid / Token CSRF invalido

Causa: La sesion expiro o las cookies fueron eliminadas.

Solucion:

  • Limpia las cookies del navegador para el dominio
  • Vuelve a iniciar sesion
  • Si el problema persiste, verifica que la configuracion de sesiones de PHP es correcta (directorio de sesiones con permisos de escritura)
● "Debe cambiar su contrasena" aparece al iniciar sesion

Causa: Es el comportamiento normal en el primer inicio de sesion. El sistema obliga a cambiar la contrasena generada automaticamente.

Solucion:

  • Ingresa una nueva contrasena en el dialogo que aparece
  • La contrasena debe ser segura (minimo 6 caracteres recomendado)
  • Despues de cambiarla, podras acceder normalmente a la aplicacion
● Contrasena de administrador perdida

Causa: Se perdio la contrasena del usuario admin y no hay forma de recuperarla desde la interfaz.

Solucion:

⚠
Este proceso eliminara todos los datos de la aplicacion (usuarios, conexiones, consultas guardadas y logs de auditoria).
  1. Accede al servidor via FTP o cPanel File Manager
  2. Elimina el archivo data/app.sqlite
  3. Navega nuevamente a la URL de la aplicacion
  4. La aplicacion recreara la base de datos con un nuevo usuario admin y nueva contrasena aleatoria
  5. Lee la nueva contrasena del archivo data/.initial_password
● PDO driver not found / Driver PDO no encontrado

Causa: La extension PDO necesaria no esta instalada o habilitada en PHP.

Solucion:

  • En cPanel: ve a Select PHP Version y habilita la extension necesaria (pdo_mysql, pdo_sqlite, etc.)
  • Si no tienes acceso a la configuracion de PHP, contacta al proveedor de hosting para solicitar la habilitacion de la extension
  • Para SQL Server en Linux, puede ser necesario instalar el driver ODBC de Microsoft. Consulta la documentacion de Microsoft para instrucciones especificas
● .htaccess no funciona / No se aplican las reglas

Causa: Apache no esta configurado para leer archivos .htaccess en ese directorio.

Solucion:

  • Contacta a tu proveedor de hosting para verificar que AllowOverride All esta habilitado para tu directorio
  • Si tienes acceso a la configuracion de Apache, agrega en el bloque <Directory> correspondiente:
<Directory "/ruta/a/public_html/phpadmin">
    AllowOverride All
</Directory>
  • Reinicia Apache despues de hacer cambios: sudo systemctl restart apache2

🌐 Problemas Reales de Deploy en Hosting Compartido

Estos problemas fueron encontrados durante despliegues reales en hosting compartido.

● La app se queda en "Cargando..." / pantalla en blanco

Causa: Error en la comunicacion con la API backend.

Solucion:

  • Abrir la consola del navegador (F12) y verificar si hay errores 404 o 500 en las peticiones a /api/
  • Si hay 404: el .htaccess no esta funcionando. Verificar que mod_rewrite este activo y que RewriteBase sea correcto
  • Si hay 500: hay un error PHP. Abrir directamente https://dominio.com/api/csrf-token en el navegador para ver el mensaje de error
● GET /phpadmin/api/csrf-token 404 (baseUrl incorrecto)

Causa: En versiones anteriores, el baseUrl del API client estaba hardcodeado con la ruta /phpadmin/.

Solucion:

  • En v1.1.0 el baseUrl se auto-detecta. Si persiste el error, limpiar la cache del navegador y el Service Worker
  • En Chrome: F12 → Application → Service Workers → Unregister
  • Hacer hard refresh: Ctrl+Shift+R
● SQLSTATE[HY000]: General error: 1 duplicate column name: must_change_password

Causa: Se subio un archivo app.sqlite del entorno local que tiene un esquema incompatible con la version desplegada.

Solucion:

  • Eliminar data/app.sqlite del hosting (via FTP o cPanel File Manager)
  • La base de datos se regenera automaticamente al siguiente acceso con el esquema correcto
  • Se creara un nuevo usuario admin con contrasena aleatoria en data/.initial_password
● "This extension requires the Microsoft ODBC Driver for SQL Server"

Causa: El hosting tiene la extension php_sqlsrv habilitada en php.ini pero no tiene el Microsoft ODBC Driver instalado en el sistema operativo.

Solucion:

  • Hosting compartido no soporta SQL Server. Solo se puede usar MySQL/MariaDB
  • La app v1.1.0 suprime este warning automaticamente y valida los drivers antes de intentar conectar
  • Si necesitas SQL Server, usa un VPS o servidor dedicado con el ODBC Driver instalado
● No puedo ver data/.initial_password

Causa: Es un archivo oculto (empieza con punto) y ademas esta protegido por .htaccess contra acceso web.

Solucion:

  • Acceder via FTP y activar la opcion de ver archivos ocultos en el cliente FTP
  • En cPanel File Manager: ir a Configuracion (esquina superior derecha) y activar "Mostrar archivos ocultos (dotfiles)"
  • Alternativa: Subir check_setup.php (disponible en la raiz del proyecto) a la raiz del hosting y abrirlo en el navegador. Muestra las credenciales y diagnostico completo
  • IMPORTANTE: Eliminar check_setup.php inmediatamente despues de usar
● Error 500 al comparar esquemas / cross-join

Causa: La extension pdo_sqlite no esta disponible en el hosting. Es necesaria para la base de datos interna que almacena resultados temporales de comparacion de esquemas y cross-join.

Solucion:

  • Verificar que pdo_sqlite esta habilitado en cPanel → Select PHP Version
  • Ejecutar check_setup.php para un diagnostico completo de extensiones y permisos
  • Si pdo_sqlite no esta disponible, contactar al proveedor de hosting para solicitarlo
